Below is a **point-wise explanation in simple language for exam writing**. Each topic has **short definitions and key points**, which are easy to **remember and write in exams**.

---

# 1. Database Management System (DBMS)

1. DBMS is **software used to create, store, manage and retrieve data** in a database.
2. It allows **multiple users to access data efficiently**.
3. It organizes data in **tables, records, and fields**.
4. It provides **security, backup, and recovery**.
5. It reduces **data redundancy and inconsistency**.

**Examples:** MySQL, Oracle, PostgreSQL.

---

# 2. File Based System

1. In a file-based system, data is stored in **separate files**.
2. Each application program has its **own data files**.
3. There is **no central database control**.
4. Data is managed by **individual programs**.

**Disadvantages**

* Data redundancy (duplicate data)
* Data inconsistency
* Difficult data access
* Poor security
* No data sharing

---

# 3. Advantages of DBMS Over File Based System

1. **Reduced Data Redundancy**
   DBMS reduces duplication of data.

2. **Data Consistency**
   Same data remains accurate everywhere.

3. **Better Data Security**
   Only authorized users can access data.

4. **Data Sharing**
   Multiple users can access the database.

5. **Backup and Recovery**
   Data can be restored after system failure.

6. **Data Integrity**
   Ensures accuracy and reliability of data.

---

# 4. Database Approach

1. Data is stored in a **centralized database**.
2. DBMS controls **data storage and access**.
3. Data can be **shared among multiple users**.
4. Reduces data redundancy.
5. Provides **data independence**.

---

# 5. Logical DBMS Architecture

1. Logical architecture describes **how data is viewed by users**.
2. It focuses on **structure and relationships of data**.
3. It hides the **physical storage details**.
4. Logical architecture is implemented using **three-level architecture**.

---

# 6. Three Level Architecture of DBMS

The DBMS architecture is divided into **three levels**.

### External Level

1. Describes **user view of database**.
2. Different users see **different views of data**.
3. Helps in **data security**.

Example:
Student sees marks but not salary of teachers.

---

### Conceptual Level

1. Describes **overall logical structure of database**.
2. Defines **tables, attributes, relationships, and constraints**.
3. Independent of physical storage.

---

### Internal Level

1. Describes **how data is physically stored in storage devices**.
2. Includes **file organization, indexing, and storage methods**.

---

# 7. Need for Three Level Architecture

1. Provides **data independence**.
2. Allows **different views for different users**.
3. Improves **data security**.
4. Makes database **easier to maintain and modify**.

---

# 8. Physical DBMS Architecture

1. Physical architecture shows **hardware components of DBMS**.
2. Includes **server, storage devices, memory, and network**.
3. DBMS software interacts with **hardware and database files**.

Flow:

User → Application → DBMS → Database Storage

---

# 9. Database Administrator (DBA)

DBA is the **person responsible for managing and controlling the database system**.

### Functions of DBA

1. Designing database structure.
2. Managing user access and permissions.
3. Ensuring database security.
4. Performing backup and recovery.
5. Monitoring database performance.
6. Maintaining data integrity.

---

# 10. Data Files

1. Data files store the **actual data records of the database**.
2. They contain **tables and records**.
3. Stored on **disk storage**.

Example
Student records stored in student data file.

---

# 11. Index

1. Index is a **data structure used to speed up data retrieval**.
2. Works like **index in a book**.
3. Helps database find data **quickly without scanning entire table**.

Example
Index on Student ID.

---

# 12. Data Dictionary

1. Data dictionary is a **repository of metadata (data about data)**.
2. It stores information about:

* Tables
* Attributes
* Data types
* Constraints
* Relationships

3. Helps DBMS **manage database structure**.

---

# 13. Types of Database

Common types of databases:

1. **Relational Database** – data stored in tables.
2. **Hierarchical Database** – data stored in tree structure.
3. **Network Database** – data stored in network structure.
4. **Object-Oriented Database** – stores objects.
5. **Distributed Database** – data stored across multiple locations.
6. **NoSQL Database** – used for large-scale data.

---

# 14. Data Models

1. Data model defines **how data is organized and structured**.
2. It describes **relationships between data elements**.

Examples:

* ER Model
* Relational Model
* Hierarchical Model

---

# 15. ER Model (Entity Relationship Model)

1. ER model is used for **database design**.
2. It represents **entities, attributes, and relationships**.

### Components

**Entity**
Real-world object.

Example: Student, Teacher.

**Attribute**
Property of an entity.

Example: Name, Age.

**Relationship**
Connection between entities.

Example: Student enrolls in Course.

---

# 16. Relational Model

1. Data is stored in **tables called relations**.
2. Each table has **rows and columns**.
3. Tables are connected using **keys**.

Example

Student Table

ID | Name | Age

---

# 17. Domain

1. Domain is the **set of valid values for an attribute**.

Example
Age → values between 0 and 120.

---

# 18. Tuple

1. Tuple is **one row in a table**.
2. Represents a **single record**.

Example

1 | Rishi | 22

---

# 19. Relation

1. Relation is **a table in relational database**.
2. It contains **tuples (rows) and attributes (columns)**.

---

# 20. Super Key

1. Super key is **a set of attributes that uniquely identifies a record**.

Example

StudentID.

---

# 21. Candidate Key

1. Candidate key is **minimal super key**.
2. It uniquely identifies records with **minimum attributes**.

Example

Roll Number.

---

# 22. Primary Key

1. Primary key is the **selected candidate key**.
2. Used to uniquely identify each record.

Rules

* Must be unique
* Cannot be NULL

Example

StudentID.

---

# 23. Foreign Key

1. Foreign key is an **attribute that links two tables**.
2. It references **primary key of another table**.

Example

CourseID in Student table referencing Course table.

---

# 24. Relational Constraints

1. Constraints are **rules applied to maintain data accuracy and consistency**.

---

# 25. Domain Constraint

1. Attribute value must belong to **valid domain**.

Example

Age cannot be negative.

---

# 26. Key Constraint

1. Primary key must be **unique**.
2. No two records can have **same primary key value**.

---

# 27. Integrity Constraint

Ensures **accuracy and consistency of data**.

Types:

1. **Entity Integrity** – Primary key cannot be NULL.
2. **Referential Integrity** – Foreign key must match primary key in another table.

---

# 28. Update Operations

Database supports three update operations:

### Insert

Add new record into table.

### Delete

Remove existing record.

### Update

Modify existing data.

---

# 29. Dealing with Constraint Violations

Constraint violation occurs when **database rules are broken**.

Example

* Duplicate primary key
* Foreign key without matching primary key

DBMS prevents these errors to **maintain data integrity**.

---

# 30. Relational Operations

Basic operations used in relational algebra.

### Selection (σ)

Select rows that satisfy condition.

Example
Students with marks > 50.

---

### Projection (π)

Select specific columns.

Example
Name and Age columns.

---

### Union (∪)

Combine records from two tables.

---

### Cartesian Product (×)

Combine every row of one table with every row of another table.

---

### Join

Combine two tables using **common attribute**.

Example
Student table joined with Course table.
